# نطاق Backend المقترح للموارد البشرية

## 1) الوضع الحالي
حاليًا موديول الموارد البشرية في الواجهة الأمامية متقسم لشرائح واضحة، وفيه أجزاء شغالة على API فعلي وأجزاء جاهزة كـ shell لحين ربط backend.

الأجزاء المرتبطة فعليًا ببيانات backend:
- الموظفين (`/hr/employees` + `/hr/employees/new`)
- ملف الموظف (`/hr/employees/:id`)
- إعدادات الموارد البشرية (Master Data): الأقسام، المسميات الوظيفية، الوظائف/المناصب (`/hr/settings`)
- المستندات (`/hr/documents`)
- السلف والخصومات (`/hr/loans`)
- المرتبات وكشوف المرتبات (`/hr/payroll`)

الأجزاء الجاهزة كـ readiness/shell فقط حتى الآن:
- الحضور والانصراف (`/hr/attendance`)
- الإجازات (`/hr/leaves`)
- العُهد (`/hr/assets`)

## 2) الصفحات الجاهزة فعليًا
الصفحات القابلة للاستخدام الآن على بيانات حقيقية:
- `/hr/employees`
- `/hr/employees/new`
- `/hr/employees/:id`
- `/hr/settings`
- `/hr/documents`
- `/hr/loans`
- `/hr/payroll`

## 3) الصفحات التي تحتاج backend قبل الاستخدام الفعلي
بناءً على الفحص الحالي، الصفحات التالية تحتاج backend قبل التشغيل الحقيقي:
- الحضور والانصراف
- الإجازات
- العُهد

## 4) أقل نطاق backend مطلوب للحضور والانصراف
MVP مقترح بدون تعقيد زائد:
- جدول/كيان سجل حضور يومي لكل موظف.
- الحقول الأساسية:
  - `employeeId`
  - `workDate`
  - `status`
  - `checkInAt`
  - `checkOutAt`
  - `notes`
  - `createdBy`
  - `updatedBy`
- APIs أساسية:
  - قائمة حسب التاريخ + بحث بالموظف.
  - Upsert حالة اليوم للموظف.
  - تسجيل حضور / تسجيل انصراف (إن كانت مناسبة كـ endpoints منفصلة).

خارج نطاق MVP الآن:
- الشيفتات المتقدمة.
- الربط مع أجهزة البصمة.
- قواعد overtime معقدة.

## 5) أقل نطاق backend مطلوب للإجازات
MVP مقترح:
- كيان أنواع إجازات بسيط.
- كيان طلبات إجازة.
- الحقول الأساسية:
  - `employeeId`
  - `leaveTypeId` (أو `leaveType` كبداية)
  - `startDate`
  - `endDate`
  - `daysCount`
  - `status`
  - `notes`
- APIs أساسية:
  - إنشاء طلب.
  - عرض الطلبات مع فلترة بسيطة.
  - اعتماد / رفض / إلغاء.

خارج نطاق MVP الآن:
- سياسات تراكم رصيد معقدة.
- Carry forward وقواعد متعددة المستويات.

## 6) أقل نطاق backend مطلوب للعُهد
MVP مقترح:
- كيان عهدة موظف.
- الحقول الأساسية:
  - `employeeId`
  - `assetName`
  - `assetType`
  - `serialNo` أو `assetCode`
  - `assignedAt`
  - `returnedAt`
  - `status`
  - `notes`
- APIs أساسية:
  - تسجيل عهدة.
  - تسليم عهدة لموظف.
  - استرداد عهدة.
  - تسجيل مفقودة / تالفة.

خارج نطاق MVP الآن:
- التكامل الكامل مع مخزون المخازن.
- دورة أصول Enterprise كاملة.

## 7) تأثير ذلك على المرتبات
- لاحقًا، بيانات الحضور والإجازات ممكن تؤثر على خصومات/بدلات المرتبات.
- الأفضل عدم عمل coupling قوي مع المرتبات قبل استقرار السجلات التشغيلية اليومية.
- صفحة المرتبات الحالية تقدر تكمل بشكلها الحالي في المرحلة الحالية.

## 8) القرار المقترح
### Option A
التوقف عند نطاق HR الحالي كـ Frontend MVP والانتقال لموديول آخر في sale-readiness.

### Option B
بدء Backend تدريجيًا بالترتيب: الحضور ثم الإجازات ثم العُهد.

### التوصية
- لو عند الشركة احتياج تشغيلي قريب لتسجيل حضور يومي: نبدأ **الحضور** أولًا في backend.
- لو الاحتياج غير عاجل: نثبّت HR الحالي كـ MVP ونكمل أولويات modules أخرى مرتبطة بالبيع والتشغيل.

## 9) خطوات التنفيذ المقترحة لو اخترنا backend
1. Attendance backend MVP.
2. ربط `/hr/attendance` مع API الحقيقي.
3. Leaves backend MVP.
4. ربط `/hr/leaves` مع API الحقيقي.
5. Assets/Custody backend MVP.
6. ربط `/hr/assets` مع API الحقيقي.
7. دمج تأثير الحضور/الإجازات/العُهد على المرتبات لاحقًا بعد استقرار السجلات.

## 10) ممنوعات
- ممنوع رجوع أي صفحة HR قديمة (Legacy).
- ممنوع صفحة HR monolithic كبيرة.
- ممنوع بيانات وهمية في الواجهة.
- ممنوع تعقيد HR enterprise مبكرًا قبل حاجة فعلية.
- ممنوع استخدام مصطلح "مسير الرواتب".
