# جرد Backend الحضور قبل التنفيذ

## 1) هل يوجد backend للحضور حاليًا؟
**النتيجة: تم تنفيذ MVP للحضور.**

بناءً على الفحص:
- `hr.controller.ts`: تمت إضافة endpoints للحضور (`GET /attendance`, `POST /attendance`, `POST /attendance/record`).
- `hr.service.ts`: تمت إضافة list/save/upsert للحضور اليومي.
- `hr.dto.ts`: تمت إضافة DTOs مخصصة للحضور.
- `database.types.ts`: تمت إضافة جدول `hr_attendance_records`.
- `migrations`: تمت إضافة migration للحضور.
- `frontend hr.api.ts` و `useHr.ts`: تمت إضافة دوال/هوكس فعلية لقراءة/حفظ سجلات الحضور.

## 2) الوضع الحالي لصفحة /hr/attendance
الصفحة الحالية متصلة ببيانات حضور فعلية عبر API.

تفصيلاً:
- تستخدم API الحضور لعرض حالة كل موظف في اليوم المحدد.
- تدعم حفظ اليوم بشكل bulk.
- تدعم تعديل حالة/مواعيد/ملاحظات لكل موظف.

## 3) نطاق الحضور المطلوب
الاتجاه المعتمد:
- البداية بـ **حضور يدوي بسيط داخل النظام**.
- تجهيز التصميم ليدعم **استيراد CSV/Excel لاحقًا**.
- **بدون** أي تكامل مباشر مع أجهزة البصمة/الأجهزة الآن.

## 4) أقل جداول/حقول مطلوبة
MVP مقترح (تصميم فقط، بدون تنفيذ الآن):

كيان: سجل حضور يومي للموظف
- `employeeId`
- `workDate`
- `status`
- `checkInAt`
- `checkOutAt`
- `source` (`manual` | `import`)
- `notes`
- `createdBy`
- `updatedBy`
- `createdAt`
- `updatedAt`

الحالات المقترحة:
- `present`
- `absent`
- `late`
- `half_day`
- `leave`
- `excused`
- `early_leave`

## 5) أقل API مطلوب
MVP APIs مقترحة (بدون تنفيذ الآن):
- قائمة الحضور حسب التاريخ + بحث بالموظف.
- حفظ/تحديث سجل حضور يوم كامل بشكل bulk.
- upsert سجل موظف واحد في يوم محدد.
- (اختياري لاحقًا) endpoint لمعاينة ملف الاستيراد.
- (اختياري لاحقًا) endpoint لاعتماد الاستيراد.

## 6) UX المطلوب
تدفق استخدام بسيط لمستخدم HR:
1. اختيار التاريخ.
2. عرض قائمة الموظفين.
3. زر "تحديد الكل حاضر".
4. تعديل الاستثناءات فقط (غائب/متأخر/بعذر...).
5. حفظ اليوم.
6. لاحقًا: استيراد CSV/Excel لنفس اليوم.

## 7) تأثير الحضور على المرتبات
- في أول مرحلة، يفضل عدم ربط الحضور بالمرتبات بشكل مباشر.
- لاحقًا، كشف المرتبات يقرأ ملخصات الحضور بعد استقرار السجلات التشغيلية.
- لا نغير منطق المرتبات الحالي الآن.

## 8) Risks / guardrails
- ممنوع أي بيانات وهمية.
- ممنوع تكامل أجهزة الحضور في المرحلة الأولى.
- ممنوع تعقيد الشيفتات/الإضافي في أول pass.
- ممنوع كسر الصفحات الحالية: الموظفين/السلف/المرتبات.
- الالتزام بصياغة عربية مهنية مناسبة للسوق المصري.

## 9) Recommended implementation order
1. attendance migration/types.
2. backend DTO/service/controller.
3. frontend API/hooks.
4. wire `/hr/attendance`.
5. tests/QA.
6. import CSV/Excel later.
